# Kazna AI Helper

Расширение для 1С:Предприятие 8.3, которое распознаёт назначение платежа в банковской выписке и заполняет расшифровку документа «Поступление на расчётный счёт» с помощью языковых моделей.

Лицензия: MIT. Платформа: 1С:Предприятие 8.3.24+.

## О проекте

Расширение добавляет автоматическую обработку поля «Назначение платежа» в конфигурациях 1С на базе БСП (Бухгалтерия предприятия и совместимые). Вместо ручного разбора текста назначения и подбора договора контрагента расширение:

1. отправляет текст назначения платежа в LLM (YandexGPT, GigaChat, GPT, Claude и другие);
2. сопоставляет ответ со списком договоров контрагента из базы 1С;
3. заполняет расшифровку платежа (договор, сумма, статья ДДС, счета расчётов) через типовой механизм 1С;
4. помечает документ флажком «требуется ручная проверка», если модель не уверена или ответ не прошёл валидацию;
5. логирует каждый запрос к ИИ с метриками токенов и времени.

Для бухгалтерий и финансистов, которые ежедневно обрабатывают десятки и сотни входящих платежей с неструктурированным назначением.

### Что даёт

- Расшифровка выписки из ручной операции превращается в фоновую задачу.
- Модель выбирает договор строго из справочника контрагента, а не придумывает его.
- Сомнительные случаи помечаются и попадают на ручную проверку.
- Каждый запрос к ИИ и ответ сохраняются в отдельном журнале.

## Архитектура

Расширение состоит из двух слоёв.

### 1. Ядро коннектора ИИ (библиотека «ИИкона», префикс `КИИ_`)

Универсальный HTTP-коннектор к языковым моделям: отправка промптов, разбор ответа, логирование, OAuth2 для GigaChat, парсер Markdown. Модули:

| Модуль | Назначение |
|---|---|
| `КИИ_КоннекторИИ` | HTTP-запросы к API провайдеров, обработка ошибок, retry |
| `КИИ_ЭкспертыИИ` | Управление «экспертами» (модель + промпт + схема ответа) |
| `КИИ_ВалидацияОтветаИИ` | Валидация JSON-ответа модели по схеме |
| `КИИ_МаркдаунПарсерКлиентСервер` | Преобразование Markdown в HTML |

Метаданные ядра:

- Справочники `КИИ_МоделиИИ`, `КИИ_ЭкспертыИИ`, `КИИ_СхемыОтветаИИ`, `КИИ_ВалидаторыОтветаИИ`.
- Регистр сведений `КИИ_ЛогЗапросовИИ`, журнал всех запросов (промпт, ответ, токены, время).
- Перечисления: `КИИ_ТипыАвторизации`, `КИИ_ТипыФорматовAPI`, `КИИ_ТипыПолейСхемыОтветаИИ`, `КИИ_КлассыРезультатаВалидацииОтветаИИ`, `КИИ_НазначенияВалидатораОтветаИИ`.
- Обработки `КИИ_ТестИИ`, `КИИ_ТестМаркдаунПарсера` для проверки и отладки.

### 2. Бизнес-логика Kazna (префикс `kh_`)

Прикладной слой, который связывает коннектор ИИ с документами 1С:

| Модуль | Назначение |
|---|---|
| `kh_АнализНазначенияПлатежа` | Полный цикл анализа текста назначения: формирует промпт со списком договоров контрагента, опрашивает модель, обрабатывает кандидатов, считает уверенность, решает «подтверждено / спорно / ручная проверка» |
| `kh_ИнтеграцияИИ` | Обёртка над ядром: привязка к конкретному эксперту «Анализ назначения платежа», формирование системного промпта с форматом ответа |
| `kh_ОбработкаПоступленияНаРасчетныйСчет` | Точки входа в документ `ПоступлениеНаРасчетныйСчет`: автозапуск при записи новых документов, ручной перезапуск с формы, дозаполнение строки расшифровки типовыми реквизитами (статья ДДС, счета расчётов, налоговые признаки) |
| `kh_ЖурналБанковскихВыписок` | Программная настройка журнала документов «Банковские выписки»: добавляет колонку «Требуется проверка» (флажок `kh_ТребуетсяРучнаяПроверкаИИ`) в список документов |
| `КонтрольСборкиKazna_AI_Helper` | Служебный модуль с функцией `НомерВерсии()` для контроля факта обновления расширения в ИБ |

## Возможности

### Автоматический анализ при записи

При записи нового документа «Поступление на расчётный счёт» (вид операции «Оплата от покупателя») расширение:

1. Берёт текст назначения платежа, контрагента и сумму.
2. Передаёт их в настроенного эксперта ИИ вместе со списком договоров контрагента (до 50).
3. Получает от модели структурированный JSON-ответ с кандидатами `{contract, amount, confidence}`.
4. Если уверенность модели выше порога эксперта и валидация пройдена, заполняет расшифровку платежа.

### Доверенное автозаполнение расшифровки

ИИ проставляет только `ДоговорКонтрагента` и `СуммаПлатежа`. Остальные реквизиты строки (счета расчётов, статья ДДС, ставка НДС, порядок отражения аванса) дозаполняются через типовой механизм `ПоступлениеНаРасчетныйСчетФормы.СвойстваСтрокРасшифровкиПлатежаСервер`, без ручных хаков метаданных. Сумма документа в шапке не пересчитывается.

### Безопасная обработка неуверенных ответов

- Модель может вернуть признак `needsManualReview`. Тогда документ помечается для ручной проверки.
- Если уверенность кандидата ниже порога эксперта, он попадает в «спорные», автозаполнение не выполняется.
- При ошибках валидации схемы расширение делает до 5 повторных запросов к модели с пояснением ошибки.
- При исчерпании повторов документ помечается «требуется ручная проверка».

### Ручной перезапуск

С формы документа можно принудительно перезапустить анализ. Старая расшифровка очищается и заполняется заново.

### Маркер в журнале выписок

В стандартном журнале «Банковские выписки» появляется колонка-флажок «Требуется проверка». Бухгалтер сразу видит документы, которые ИИ не смог обработать автоматически.

### Полное логирование

Все запросы к ИИ пишутся в регистр `КИИ_ЛогЗапросовИИ` с привязкой к документу-инициатору, номером попытки, текстом промпта, ответом модели, количеством токенов и временем выполнения.

### Защита от случайного запуска

Анализ не запускается при загрузке данных через обмен (`ОбменДанными.Загрузка = Истина`) и при явной установке служебного свойства `kh_ПропуститьАнализИИ`. Это удобно для массовой загрузки тестовых выписок.

## Поддерживаемые провайдеры ИИ

Расширение работает с любым провайдером, поддерживаемым ядром «ИИкона»:

| Провайдер | Формат API | Статус |
|---|---|---|
| YandexGPT | Yandex Cloud | поддерживается |
| GigaChat (Сбер) | OAuth2 + OpenAI | поддерживается |
| OpenAI (GPT-4, GPT-5.x) | OpenAI Compatible | поддерживается |
| Anthropic (Claude) | Anthropic | поддерживается |
| Google (Gemini) | Google AI | поддерживается |
| DeepSeek | OpenAI Compatible | поддерживается |
| Ollama (локальные модели) | OpenAI Compatible | поддерживается |
| Другие OpenAI-compatible (OpenRouter, Together, Mistral, Qwen, Groq) | OpenAI Compatible | поддерживается |

API-ключи хранятся в защищённом хранилище 1С и не попадают в исходники расширения.

## Установка

### Требования

- 1С:Предприятие 8.3.24+ (разрабатывалось на 8.3.27).
- БСП 3.1.10+.
- Конфигурация с документом `ПоступлениеНаРасчетныйСчет` и табличной частью `РасшифровкаПлатежа` (Бухгалтерия предприятия 3.0 и совместимые).
- Доступ в интернет к API выбранного провайдера ИИ, кроме локальных моделей Ollama.

### Шаги

1. Склонируйте репозиторий или скачайте исходники.
2. В Конфигураторе: «Конфигурация → Расширения → Добавить из файлов», укажите корень каталога `Kazna_AI_Helper` (где лежит `Configuration.xml`).
3. Снимите флаг «Безопасный режим» при необходимости (для HTTP-запросов к API).
4. Выполните «Обновление конфигурации базы данных» (`F7`).
5. В справочнике `КИИ_МоделиИИ` создайте модель нужного провайдера и укажите API-ключ.
6. В справочнике `КИИ_ЭкспертыИИ` создайте эксперта для назначения «Анализ назначения платежа поступления на Р/С»:
   - укажите настроенную модель;
   - заполните описание эксперта (системный промпт);
   - при необходимости задайте порог уверенности кандидата.
7. Проверьте установку: в режиме Предприятия выполните в табло `КонтрольСборкиKazna_AI_Helper.НомерВерсии()`. Должно вернуться значение, указанное в исходниках модуля `КонтрольСборкиKazna_AI_Helper`.

## Как это работает

### Автоматический режим

1. Пользователь или загрузка выписки клиент-банком создаёт документ «Поступление на расчётный счёт» с видом операции «Оплата от покупателя».
2. При записи расширение планирует отложенный анализ.
3. Сразу после записи выполняется запрос к ИИ, валидация ответа, дозаполнение расшифровки.
4. Документ перезаписывается (без перепроведения) с заполненной расшифровкой и снятым или поставленным флажком ручной проверки.

### Ручной режим

Если автозаполнение не сработало (низкая уверенность, ошибка API, изменённая расшифровка), на форме документа доступна команда перезапуска анализа. Расшифровка очищается и заполняется заново по актуальному ответу модели.

## Программный интерфейс (для доработок)

### Анализ назначения платежа

```bsl
// Полный цикл анализа без записи документа.
// НазначениеПлатежа, Контрагент и СуммаДокумента берутся из документа.
Результат = kh_АнализНазначенияПлатежа.ВыполнитьАнализ(
    НазначениеПлатежа,
    Контрагент,
    СуммаДокумента,
    ДокументСсылка);

Если Результат.МожноАвтозаполнять Тогда
    Для Каждого Кандидат Из Результат.ПодтвержденныеКандидаты Цикл
        // Кандидат.Договор: СправочникСсылка.ДоговорыКонтрагентов
        // Кандидат.Сумма:   Число
    КонецЦикла;
ИначеЕсли Результат.НужнаРучнаяПроверка Тогда
    Сообщить(Результат.Сообщение);
КонецЕсли;
```

### Структура результата анализа (`kh_АнализНазначенияПлатежа.НовыйРезультатАнализа`)

| Поле | Тип | Описание |
|---|---|---|
| `ЭтоОшибкаAPI` | Булево | Ошибка на стороне HTTP-запроса к модели |
| `ТекстОшибкиAPI` | Строка | Текст ошибки API |
| `ИтогВалидации` | Структура | Подробный итог валидации ответа по схеме |
| `ПодтвержденныеКандидаты` | Массив | Кандидаты с договором, суммой и пройденной валидацией |
| `СпорныеКандидаты` | Массив | Кандидаты с замечаниями валидации или низкой уверенностью |
| `МодельЗапросилаРучнуюПроверку` | Булево | Модель вернула `needsManualReview = true` |
| `НужнаРучнаяПроверка` | Булево | Агрегированный признак для документа |
| `МожноАвтозаполнять` | Булево | Есть подтверждённые кандидаты и нет блокирующих ошибок |
| `ПовторитьЗапрос` | Булево | Сигнал повторить запрос к модели |
| `Сообщение` | Строка | Пользовательское сообщение о результате |

### Прямой запрос к модели (через ядро «ИИкона»)

```bsl
Модель = Справочники.КИИ_МоделиИИ.НайтиПоНаименованию("YandexGPT 5 Pro");
Ответ = КИИ_КоннекторИИ.ЗапросКМодели(Модель, "Привет! Как дела?");

Если Не Ответ.ЭтоОшибка Тогда
    Сообщить(Ответ.ТекстОтвета);
    Сообщить("Токенов: " + Ответ.ВсегоТокенов);
КонецЕсли;
```

## Получение API-ключей

### YandexGPT

1. Зарегистрируйтесь в Yandex Cloud.
2. Создайте сервисный аккаунт и получите API-ключ.
3. В модели укажите ваш Folder ID.

### GigaChat (Сбер)

1. Зарегистрируйтесь на developers.sber.ru.
2. Получите `ClientId` и `ClientSecret`.
3. В 1С укажите в формате `ClientId:ClientSecret`. Расширение автоматически обменяет их на OAuth2-токен.

### OpenAI / Anthropic / Google / DeepSeek

Создайте ключ в личном кабинете соответствующего провайдера и укажите его в карточке модели.

### Ollama (локально)

Ключ не требуется. Запустите Ollama локально (`localhost:11434`) и создайте модель с форматом `OpenAI_Compatible`.

## Качество и совместимость

- Код проверен статическими анализаторами BSL Language Server.
- Имена новых объектов используют префикс `kh_` для прикладного кода и `КИИ_` для ядра коннектора.
- Соблюдены стандарты разработки 1С: запросы параметризованы, доступ к реквизитам ссылок через `ОбщегоНазначения.ЗначенияРеквизитовОбъекта`, без точечной нотации.

### Совместимость

Расширение рассчитано на конфигурации с документом `ПоступлениеНаРасчетныйСчет` и типовым механизмом расшифровки платежа:

- 1С:Бухгалтерия предприятия 3.0 (релизы с поддержкой БСП 3.1.10+).
